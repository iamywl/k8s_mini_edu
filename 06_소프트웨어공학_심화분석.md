## 1. 결합도/응집도 분석 (Coupling & Cohesion Analysis)

> **기법 설명**: 모듈 간의 **결합도(Coupling)**는 낮을수록, 모듈 내부의 **응집도(Cohesion)**는 높을수록 좋은 설계임.

### 1.1 Kubernetes의 결합도 분석

| 컴포넌트 쌍 | 결합 유형 | 수준 | 설명 | 인터페이스 소스 위치 |
|-------------|-----------|------|------|---------------------|
| apiserver ↔ etcd | 데이터 결합 | **낮음 (좋음)** | `storage.Interface`(interfaces.go 169행)를 통해서만 통신함. etcd3 구현체(store.go 80행)는 이 인터페이스 뒤에 숨겨져 있음 | `apiserver/pkg/storage/interfaces.go` |
| controller-manager ↔ apiserver | 데이터 결합 | **낮음 (좋음)** | REST API + Watch를 통해서만 통신함. `SharedInformer`(shared_informer.go 143행)가 Watch를 캐싱하여 API 서버 부하를 줄임 | `client-go/tools/cache/shared_informer.go` |
| kubelet ↔ containerd | 스탬프 결합 | **낮음 (좋음)** | `RuntimeService` 인터페이스(services.go 112행)를 통해 gRPC로 통신함. containerd를 CRI-O 등 다른 런타임으로 교체 가능함 | `cri-api/pkg/apis/services.go` |
| scheduler ↔ apiserver | 데이터 결합 | **낮음 (좋음)** | `Reflector.ListAndWatchWithContext()`(reflector.go 472행)를 통해 데이터를 받아옴 | `client-go/tools/cache/reflector.go` |
| pkg/controller/* 간 | 없음 | **매우 낮음 (좋음)** | 38개 컨트롤러가 서로 직접 통신하지 않음. 모두 API 서버를 통해 간접적으로 상호작용함. 공유 인프라는 `controller_utils.go`에 있음 | `pkg/controller/controller_utils.go` |

### 1.2 Kubernetes의 응집도 분석

| 모듈 | 응집도 유형 | 수준 | 설명 |
|------|------------|------|------|
| `pkg/controller/deployment` | 기능적 응집 | **높음 (좋음)** | Deployment 리소스의 관리라는 단일 기능에만 집중함. `DeploymentController`(67행), `syncDeployment()`(589행), `reconcileNewReplicaSet()`(rolling.go 68행) 모두 Deployment 전용임 |
| `pkg/controller/replicaset` | 기능적 응집 | **높음 (좋음)** | ReplicaSet 관리 전용. `syncReplicaSet()`(707행)과 `manageReplicas()`(601행)가 핵심 로직임 |
| `pkg/kubelet` | 순차적 응집 | **보통** | 파드 생명주기라는 하나의 주제이지만 46개 서브패키지로 범위가 넓음 |
| `staging/apimachinery` | 기능적 응집 | **높음 (좋음)** | API 객체의 직렬화/변환이라는 단일 목적에 집중함 |
| `staging/client-go` | 통신적 응집 | **높음 (좋음)** | API 서버와의 통신이라는 단일 목적에 집중함. `SharedInformer`, `Reflector`, `ListWatch` 모두 이 목적에 기여함 |
| `hack/` | 시간적 응집 | **보통** | "개발 시 사용하는 스크립트"라는 공통점이 있지만, 빌드/검증/코드생성 등 기능은 다양함 |

### 1.3 설계 품질 요약

```
[Kubernetes의 설계 강점]

1. API 서버 중심 통신 (Hub-and-Spoke)
   → 컴포넌트 간 직접 의존성을 제거하여 결합도를 최소화함
   → DefaultBuildHandlerChain() (config.go 1028행)이 모든 요청의 관문임

2. 인터페이스 기반 추상화
   → storage.Interface (169행): etcd를 추상화
   → RuntimeService (112행): 컨테이너 런타임을 추상화
   → Authorizer (83행): 인가 방식을 추상화
   → 구현체를 교체 가능하게 함

3. 컨트롤러별 독립 패키지
   → 38개 컨트롤러가 서로 의존하지 않아 독립적으로 개발/테스트 가능함
   → 공통 패턴만 controller_utils.go에서 공유함

4. 스테이징 모듈 분리
   → 31개 라이브러리를 독립 모듈로 분리하여 외부에서도 재사용 가능함
   → .import-restrictions 파일로 역방향 의존성을 차단함
```

## 2. 상태 다이어그램 분석 (State Diagram)

> **기법 설명**: 리소스의 생명주기를 **상태 전이 다이어그램(State Transition Diagram)**으로 분석하는 기법임.

### 2.1 파드(Pod) 상태 전이

```
                    ┌──────────┐
                    │ Pending  │ ← 파드가 생성되었지만 아직 스케줄링되지 않은 상태임
                    └────┬─────┘
                         │ 스케줄러가 노드를 배정함
                         │ (schedule_one.go — ScheduleOne() 64행)
                         ▼
                    ┌──────────┐
                    │ Running  │ ← 최소 하나의 컨테이너가 실행 중인 상태임
                    └──┬───┬───┘   (kubelet이 CRI로 컨테이너를 시작한 후)
                       │   │
            성공 완료  │   │  실패 발생
                       ▼   ▼
                ┌──────────┐ ┌──────────┐
                │Succeeded │ │  Failed  │
                │          │ │          │
                │모든 컨테 │ │하나 이상 │
                │이너가 성 │ │의 컨테이 │
                │공적으로  │ │너가 실패 │
                │완료된 상 │ │한 상태임 │
                │태임      │ │          │
                └──────────┘ └──────────┘

                    ┌──────────┐
                    │ Unknown  │ ← 노드와 통신이 안 되는 상태임
                    └──────────┘

[관련 소스]: pkg/kubelet/status/ — 파드 상태 매니저
[타입 정의]: staging/src/k8s.io/api/core/v1/types.go — PodPhase 타입
```

### 2.2 디플로이먼트(Deployment) 롤아웃 상태 전이

```
┌──────────────┐
│  Progressing │ ← 새 ReplicaSet이 생성되고 파드가 증가 중인 상태임
└──────┬───────┘   (reconcileNewReplicaSet() — rolling.go 68행)
       │ 모든 파드가 원하는 버전으로 업데이트됨
       ▼
┌──────────────┐
│  Available   │ ← 원하는 수의 파드가 모두 실행 중인 상태임
└──────┬───────┘   (syncDeploymentStatus() — sync.go 480행)
       │ 실패 조건 발생 (타임아웃, 리소스 부족 등)
       ▼
┌──────────────┐
│ Failed/      │ ← 롤아웃이 실패한 상태임
│ Degraded     │   → 롤백이 필요할 수 있음
└──────────────┘

[관련 소스]: pkg/controller/deployment/
  - syncDeployment() (deployment_controller.go 589행) — 진입점
  - reconcileNewReplicaSet() (rolling.go 68행) — 새 RS 스케일업
  - reconcileOldReplicaSets() (rolling.go 86행) — 구 RS 스케일다운
```

                                              └────┬─────┘
                                                   │
                                    kubelet 응답 없음 │
                                                   ▼
                                              ┌──────────┐
                                              │ NotReady │
                                              │ (비정상) │
                                              └────┬─────┘
                                                   │
                                     일정 시간 경과 │
                                                   ▼
                                              ┌──────────┐
                                              │ Unknown  │
                                              │ (통신불가)│
                                              └──────────┘

[관련 소스]: pkg/controller/nodelifecycle/ — 노드 상태 모니터링 컨트롤러

### 4.2 상세 유스케이스: "애플리케이션 배포"

```
유스케이스명: 애플리케이션 배포
주 액터: 개발자
사전 조건: Kubernetes 클러스터가 동작 중이고, kubectl이 설정되어 있음

기본 흐름 (관련 소스 코드):
  1. 개발자가 Deployment YAML을 작성함
  2. kubectl apply -f deployment.yaml 명령을 실행함
     → staging/src/k8s.io/kubectl/ (kubectl 핵심 로직)
  3. API 서버가 요청을 인증하고 인가함
     → WithAuthentication() (config.go 1069행)
     → WithAuthorization() (config.go 1032행)
  4. 어드미션 컨트롤러가 요청을 검증/변형함
     → chainAdmissionHandler.Admit() (chain.go 31행)
  5. Deployment 컨트롤러가 ReplicaSet을 생성함
     → syncDeployment() (deployment_controller.go 589행)
  6. ReplicaSet 컨트롤러가 파드를 생성함
     → syncReplicaSet() (replica_set.go 707행)
     → manageReplicas() (replica_set.go 601행)
  7. 스케줄러가 파드를 노드에 배치함
     → ScheduleOne() (schedule_one.go 64행)
     → schedulingCycle() (schedule_one.go 174행)
  8. kubelet이 CRI로 컨테이너를 실행함
     → RunPodSandbox() (remote_runtime.go 212행)
     → CreateContainer() (remote_runtime.go 339행)
     → StartContainer() (remote_runtime.go 370행)
  9. 개발자가 kubectl get pods로 상태를 확인함

대안 흐름:
  4a. 이미지를 Pull할 수 없는 경우:
      → 파드 상태가 ImagePullBackOff가 됨
      → 개발자에게 이미지 이름/레지스트리 확인을 요청함

  5a. 리소스가 부족한 경우:
      → 파드 상태가 Pending으로 유지됨
      → 스케줄러가 적절한 노드를 찾을 때까지 대기함
      → PostFilter(preemption)가 실행될 수 있음
        (schedule_one.go 287행)

사후 조건: 지정된 수의 파드가 Running 상태가 됨
```

## 5. SOLID 원칙 적용 분석

> **기법 설명**: 객체지향 설계의 5대 원칙(SOLID)이 Kubernetes에 어떻게 적용되어 있는지 분석하는 것임.

### 5.1 단일 책임 원칙 (SRP: Single Responsibility Principle)

| 적용 사례 | 설명 | 소스 위치 |
|-----------|------|-----------|
| 각 컨트롤러 | Deployment 컨트롤러는 Deployment만 관리하고, Job 컨트롤러는 Job만 관리함. 각각 하나의 리소스 타입에 대한 책임만 가짐 | `pkg/controller/deployment/` (DeploymentController 67행), `pkg/controller/job/` (Controller 84행) |
| kubelet의 서브패키지 | 볼륨 관리(`volume/`), 프로브(`prober/`), 이미지 관리(`images/`) 등이 각각 분리되어 하나의 책임만 담당함 | `pkg/kubelet/` (46개 서브패키지) |
| build/ vs hack/ | 빌드/릴리즈 스크립트(build/)와 개발 도구(hack/)가 명확히 분리됨 | `build/` vs `hack/` |

### 5.2 개방-폐쇄 원칙 (OCP: Open/Closed Principle)

| 적용 사례 | 설명 | 소스 위치 |
|-----------|------|-----------|
| 스케줄러 플러그인 | 스케줄러는 플러그인 프레임워크를 통해 확장 가능함. 기존 코드를 수정하지 않고 새로운 스케줄링 알고리즘을 추가할 수 있음 | `Plugin` 인터페이스 (interface.go 401행), `FilterPlugin` (505행), `ScorePlugin` (582행), `BindPlugin` (656행) |
| 어드미션 웹훅 | 외부 웹훅을 통해 어드미션 로직을 확장할 수 있음. API 서버 코드를 변경하지 않아도 됨 | `MutationInterface.Admit()` (129행), `ValidationInterface.Validate()` (138행) |
| CRI/CSI/CNI | 표준 인터페이스를 통해 컨테이너 런타임, 스토리지, 네트워크 구현체를 교체할 수 있음 | `RuntimeService` (services.go 112행) |

### 5.3 리스코프 치환 원칙 (LSP: Liskov Substitution Principle)

| 적용 사례 | 설명 | 소스 위치 |
|-----------|------|-----------|
| 컨테이너 런타임 | CRI 인터페이스를 구현한 어떤 런타임이든(containerd, CRI-O 등) kubelet에서 교체하여 사용할 수 있음 | `RuntimeService` 인터페이스 (services.go 112행), `remoteRuntimeService` 구현 (remote_runtime.go 47행) |
| 스토리지 백엔드 | `storage.Interface`를 구현한 어떤 스토리지든(etcd3 등) 동일하게 사용할 수 있음 | `Interface` (interfaces.go 169행), `store` 구현 (etcd3/store.go 80행), 컴파일 타임 확인 (100행): `var _ storage.Interface = (*store)(nil)` |
| 인가 방식 | `Authorizer` 인터페이스를 구현한 어떤 인가기든(RBAC, Node, Webhook) 교체 가능함 | `Authorizer` (interfaces.go 83행), `RBACAuthorizer` (rbac.go 50행), `NodeAuthorizer` (node_authorizer.go 63행) |

### 5.4 인터페이스 분리 원칙 (ISP: Interface Segregation Principle)

| 적용 사례 | 설명 | 소스 위치 |
|-----------|------|-----------|
| 스케줄러 플러그인 인터페이스 | 작은 인터페이스로 분리되어 필요한 것만 구현하면 됨 | `FilterPlugin` (505행), `ScorePlugin` (582행), `PreFilterPlugin` (476행), `PostFilterPlugin` (534행), `ReservePlugin` (599행), `BindPlugin` (656행), `PermitPlugin` (643행) — 각각 독립 인터페이스임 |
| staging 모듈 분리 | 31개의 독립 모듈로 분리되어, 필요한 모듈만 import하여 사용할 수 있음 | `go.work` — 32개 모듈 선언 |
| CRI 서브 인터페이스 | `RuntimeService`가 `ContainerManager`(34행), `PodSandboxManager`(67행), `ContainerStatsManager`(93행)을 조합함 | `cri-api/pkg/apis/services.go` |

### 5.5 의존성 역전 원칙 (DIP: Dependency Inversion Principle)

| 적용 사례 | 설명 | 소스 위치 |
|-----------|------|-----------|
| Storage Interface | apiserver는 구체적인 etcd 구현이 아닌 `storage.Interface`에 의존함. etcd를 다른 저장소로 교체 가능함 | `Interface` (interfaces.go 169행) vs `store` (etcd3/store.go 80행) |
| CRI Interface | kubelet은 구체적인 containerd가 아닌 `RuntimeService` 인터페이스에 의존함 | `RuntimeService` (services.go 112행) vs `remoteRuntimeService` (remote_runtime.go 47행) |
| FeatureGate | 컴포넌트들은 `FeatureGate` 인터페이스(144행)에 의존하고, 구체 구현은 `featureGate` 구조체(261행)임 | `component-base/featuregate/feature_gate.go` |

### 6.3 컨트롤러 패턴의 실제 코드 구조

모든 컨트롤러는 공식 인터페이스 없이 **동일한 패턴**을 따름:

```
[공통 패턴 — 실제 Go 구조]

type XxxController struct {
    syncHandler func(ctx context.Context, key string) error  // 테스트 주입용
    queue       workqueue.TypedRateLimitingInterface[string]  // 작업 큐
    xxxLister   xxxlisters.XxxLister                          // Informer 캐시
    xxxSynced   cache.InformerSynced                          // 초기 동기화 완료 여부
}

func (c *XxxController) Run(ctx context.Context, workers int) {
    // 1. Informer 캐시가 동기화될 때까지 대기
    cache.WaitForCacheSync(ctx.Done(), c.xxxSynced)
    // 2. workers 수만큼 고루틴 시작
    for i := 0; i < workers; i++ {
        go wait.UntilWithContext(ctx, c.worker, time.Second)
    }
}

func (c *XxxController) worker(ctx context.Context) {
    for c.processNextWorkItem(ctx) {}
}

func (c *XxxController) processNextWorkItem(ctx context.Context) bool {
    key, quit := c.queue.Get()     // 큐에서 항목 꺼냄
    if quit { return false }
    defer c.queue.Done(key)
    err := c.syncHandler(ctx, key) // reconcile 실행
    // 에러 시 큐에 다시 추가 (재시도)
}
```

## 8. 테스트 피라미드 분석 (Test Pyramid)

> **기법 설명**: 테스트 전략을 피라미드 형태로 분석하는 기법임. 아래층이 빠르고 많고, 위로 갈수록 느리고 적음.

```
                    △
                   ╱ ╲
                  ╱   ╲
                 ╱ E2E ╲         ← 전체 클러스터 테스트
                ╱  Tests╲          test/e2e/ (27개 디렉토리)
               ╱─────────╲        실행 시간: 시간 단위
              ╱           ╲
             ╱ Integration ╲     ← 컴포넌트 간 상호작용 테스트
            ╱    Tests      ╲      test/integration/ (64개 디렉토리)
           ╱─────────────────╲     실행 시간: 분 단위
          ╱                   ╲
         ╱    Unit Tests       ╲ ← 함수/메서드 단위 테스트
        ╱                       ╲   pkg/ 전체에 분포
       ╱─────────────────────────╲   실행 시간: 초 단위
      ╱                           ╲
     ╱   Static Analysis           ╲ ← 코드 실행 없이 분석
    ╱   (verify-*.sh 스크립트들)     ╲   hack/verify-*.sh (50개 이상)
   ╱─────────────────────────────────╲   실행 시간: 초~분
  ╱___________________________________╲

```

### 테스트 유형별 실행 방법

| 레벨 | 실행 명령 | 스크립트 | 실행 시간 |
|------|-----------|----------|-----------|
| 정적 분석 | `make verify` | `hack/make-rules/verify.sh` | 수 분 |
| 유닛 테스트 | `make test` | `hack/make-rules/test.sh` (287행, KUBE_TIMEOUT=180s) | 수십 분 |

## 9. 복잡도 관리 전략

> Kubernetes가 거대한 코드베이스의 복잡도를 어떻게 관리하는지 분석한 것임.

### 9.1 복잡도 관리 기법

| 기법 | 적용 방식 | 효과 |
|------|-----------|------|
| **모노레포 + 모듈 분리** | 하나의 저장소에 32개 독립 모듈을 관리함 (`go.work`) | 코드 일관성 유지 + 독립적 릴리즈 가능 |
| **Go 워크스페이스** | `go.work`로 32개 모듈을 하나의 개발 환경으로 통합함 (`go 1.25.0`) | 로컬 개발 시 모듈 간 참조가 자연스러움 |
| **코드 자동 생성** | `hack/update-codegen.sh` (1037행, 15개 생성 함수)로 반복 코드를 자동 생성함 | 수작업 오류 방지, 일관성 보장 |
| **OWNERS 파일** | 각 디렉토리에 코드 소유자를 지정함 (approvers, reviewers, labels) | 리뷰 책임을 분산, 전문성 기반 관리 |
| **기능 게이트 (Feature Gates)** | 새 기능을 플래그로 제어함 | 안전한 기능 롤아웃, 단계적 활성화 |
| **API 버전 관리** | v1, v1beta1, v1alpha1 단계별 성숙도 관리 | 호환성 유지하면서 API를 발전시킴 |

### 9.2 기능 게이트 (Feature Gate) 시스템

```
[기능 게이트의 타입 계층]

FeatureGate 인터페이스 (feature_gate.go 144행)
  └── MutableFeatureGate (161행)
        └── MutableVersionedFeatureGate (203행)
              └── featureGate 구조체 (261행) — 구체 구현

[핵심 메서드]
  Enabled(key Feature) bool        — 기능 활성화 여부 확인
  Add(features map[Feature]FeatureSpec) — 기능 등록
  SetFromMap(m map[string]bool)    — 커맨드라인에서 설정
  OverrideDefault(name Feature, ...)— 기본값 재정의
```

```
[기능 게이트의 생명주기]
정의 파일: pkg/features/kube_features.go

Alpha (v1alpha1)
  │  - Feature 상수 정의 (41행~)
  │  - 기본값: false
  │  - defaultVersionedKubernetesFeatureGates (1094행)에 등록:
  │    MyFeature: {
  │      {Version: "1.18", Default: false, PreRelease: Alpha},
  │    }
  ▼
Beta (v1beta1)
  │  - 기본값: true (또는 false)
  │    MyFeature: {
  │      ...
  │      {Version: "1.24", Default: true, PreRelease: Beta},
  │    }
  ▼
GA (General Availability, v1)
  │  - 기본값: 항상 활성화, LockToDefault: true
  │    MyFeature: {
  │      ...
  │      {Version: "1.33", Default: true, PreRelease: GA, LockToDefault: true},
  │    }
  ▼
Deprecated → Removed
  - 더 이상 사용되지 않는 기능임

[기능 간 의존성]
  defaultKubernetesFeatureGateDependencies (2115행):
    MyFeature: [DependsOnFeature1, DependsOnFeature2]

[초기화]
  init() 함수 (2520행):
    utilfeature.DefaultMutableFeatureGate에 모든 Feature 등록

[관련 스크립트]:
  hack/update-featuregates.sh — 기능 게이트 메타데이터 업데이트
  hack/verify-featuregates.sh — 기능 게이트 규칙 검증
```

## 10. 초보자를 위한 종합 요약

### "Kubernetes가 잘 만들어진 이유"

```
1. 느슨한 결합 (Loose Coupling)
   → 컴포넌트들이 API 서버를 통해 간접적으로만 통신함
   → 핵심 인터페이스: storage.Interface, RuntimeService, Authorizer
   → 하나가 고장나도 다른 것에 영향을 최소화함

2. 높은 응집도 (High Cohesion)
   → 각 컨트롤러/모듈이 하나의 명확한 역할만 담당함
   → 예: DeploymentController.syncDeployment() → Deployment 전용
   → 코드를 찾고 이해하기 쉬움

3. 인터페이스 기반 설계
   → CRI (RuntimeService 112행): 컨테이너 런타임 추상화
   → Storage (Interface 169행): 저장소 추상화
   → FeatureGate (144행): 기능 플래그 추상화
   → "플러그 앤 플레이" 방식임

4. 선언적 모델 (Declarative Model)
   → "이렇게 해라" 대신 "이런 상태가 되어야 한다"를 선언함
   → Reconciliation Loop (syncHandler 패턴)가 자동으로 수렴함
   → Deployment: 589행, ReplicaSet: 707행, Job: 821행

5. 자동화된 품질 관리
   → 50개 이상의 검증 스크립트가 코드 품질을 자동으로 확인함
   → 코드 생성 (update-codegen.sh 1037행)이 반복 작업을 자동화함
   → .import-restrictions 파일이 의존성 규칙을 강제함
```
