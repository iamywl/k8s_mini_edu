> **사용 기법**: 소프트웨어 공학의 **4+1 뷰 모델(Architectural Views)** 및 **컴포넌트 다이어그램** 기법을 활용하여 Kubernetes의 전체 구조를 분석한 문서임.

## 2. 프로세스 뷰 (Process View)

> **설명**: 각 컴포넌트가 어떤 순서로 시작되고, 어떻게 상호작용하는지를 보여주는 뷰임.

### 2.1 클러스터 시작 순서

```
[1단계] etcd 시작
    │   → 분산 키-값 저장소가 먼저 준비되어야 함
    │   → 모든 클러스터 상태를 여기에 저장함
    ▼
[2단계] kube-apiserver 시작
    │   → etcd에 연결함
    │   → REST API 엔드포인트를 노출함
    │   → 인증/인가 모듈을 초기화함
    ▼
[3단계] kube-controller-manager 시작
    │   → API 서버에 연결함
    │   → 38개의 컨트롤러 루프를 시작함
    │   → 클러스터의 "현재 상태"를 "원하는 상태"로 맞추는 기능을 함
    ▼
[4단계] kube-scheduler 시작
    │   → API 서버에 연결함
    │   → 스케줄 되지 않은 파드를 감시(Watch)함
    │   → 적절한 노드에 파드를 배치하는 기능을 함
    ▼
[5단계] kubelet 시작 (각 워커 노드)
    │   → API 서버에 등록함
    │   → 해당 노드에 배정된 파드를 실행함
    │   → 컨테이너 런타임(containerd)과 통신함
    ▼
[6단계] kube-proxy 시작 (각 워커 노드)
        → 서비스 네트워킹 규칙을 설정함
        → 클러스터 내부 DNS와 로드밸런싱을 지원함
```

### 2.2 파드 생성 과정 (요청 흐름)

```
사용자: kubectl apply -f pod.yaml
    │
    ▼
[kubectl] → pod.yaml을 읽어서 API 요청을 생성함
    │
    ▼ (HTTPS REST API)
[kube-apiserver]
    ├── 1. 인증(Authentication): "이 사용자가 누구인지 확인하는 기능을 함"
    ├── 2. 인가(Authorization): "이 사용자가 이 작업을 할 수 있는지 확인하는 기능을 함"
    ├── 3. 어드미션(Admission): "요청을 변경하거나 거부하는 기능을 함"
    └── 4. etcd에 파드 정보를 저장함 (status: Pending)
    │
    ▼ (Watch 이벤트)
[kube-scheduler]
    ├── 1. Pending 상태의 파드를 감지함
    ├── 2. 모든 노드의 리소스를 확인함
    ├── 3. 스코어링 알고리즘으로 최적 노드를 선택함
    └── 4. 파드를 특정 노드에 바인딩함 (etcd 업데이트)
    │
    ▼ (Watch 이벤트)
[kubelet] (선택된 노드)
    ├── 1. 자기 노드에 바인딩된 파드를 감지함
    ├── 2. 컨테이너 이미지를 Pull함
    ├── 3. containerd에 컨테이너 생성을 지시함
    ├── 4. 볼륨을 마운트함
    ├── 5. 네트워크를 설정함
    └── 6. 컨테이너를 실행하고 상태를 API 서버에 보고함
    │
    ▼ (Watch 이벤트)
[kube-proxy]
    └── 서비스와 연결된 파드라면 네트워킹 규칙을 업데이트함
```

## 5. 시나리오 뷰 (Scenarios / Use Cases)

### 5.1 핵심 유스케이스

| 유스케이스 | 관련 컴포넌트 | 설명 |
|-----------|--------------|------|
| 애플리케이션 배포 | kubectl → apiserver → scheduler → kubelet | 사용자가 YAML 매니페스트를 제출하면 파드가 생성되는 기능을 함 |
| 오토 스케일링 | controller-manager → apiserver → scheduler → kubelet | 부하에 따라 파드 수를 자동 조절하는 기능을 함 |
| 서비스 디스커버리 | kube-proxy, CoreDNS | 파드 간 통신을 서비스 이름으로 가능하게 하는 기능을 함 |
| 롤링 업데이트 | controller-manager (Deployment) | 무중단으로 새 버전을 배포하는 기능을 함 |
| 자가 치유 | controller-manager, kubelet | 장애 파드를 자동 재시작하는 기능을 함 |
| 스토리지 관리 | controller-manager (PV/PVC) → kubelet | 영구 스토리지를 파드에 연결하는 기능을 함 |

### 5.2 자가 치유(Self-Healing) 시나리오 상세

```
[정상 상태]
  ReplicaSet: replicas=3, 현재 파드 3개 실행 중

[장애 발생]
  워커 노드 2가 다운됨 → 파드 1개 소실

[감지]
  kubelet(노드2)이 응답하지 않음
    → node-lifecycle 컨트롤러가 노드를 NotReady로 마킹함

[복구]
  ReplicaSet 컨트롤러가 감지함:
    "원하는 상태(3개) ≠ 현재 상태(2개)"
    → 새 파드 생성 요청을 API 서버에 보냄
    → 스케줄러가 건강한 노드를 선택함
    → kubelet이 새 파드를 생성함

[결과]
  다시 파드 3개가 실행되는 상태로 복원됨
```

## 7. 디자인 패턴 분석

> Kubernetes 소스 코드에서 사용된 주요 소프트웨어 디자인 패턴을 분석함.

### 7.1 컨트롤러 패턴 (Reconciliation Loop)

Kubernetes의 핵심 패턴임. **모든 컨트롤러**가 이 패턴을 사용함.

```
무한 루프:
    1. 현재 상태(Current State)를 관찰함
    2. 원하는 상태(Desired State)와 비교함
    3. 차이가 있으면 현재 상태를 원하는 상태로 변경함
    4. 1번으로 돌아감
```

```
[Desired State]          [Current State]
  replicas: 3       ?=     running: 2
       │                       │
       └───────┬───────────────┘
               │
        차이 발견! (3 ≠ 2)
               │
               ▼
     파드 1개 추가 생성
```

**소스 위치**: `pkg/controller/` (38개 서브디렉토리에서 이 패턴 사용)

### 7.2 팩토리 패턴 (Factory Pattern)

API 서버에서 리소스 종류에 따라 적절한 핸들러를 생성하는 패턴임.

**소스 위치**: `pkg/registry/`

### 7.3 옵저버 패턴 (Watch Mechanism)

컴포넌트들이 API 서버의 변경 사항을 **실시간으로 감시(Watch)**하는 패턴임.

```
API 서버가 이벤트를 발행함
    │
    ├── scheduler가 "새 파드 생성" 이벤트를 수신함
    ├── controller-manager가 "파드 상태 변경" 이벤트를 수신함
    └── kubelet이 "노드에 파드 배정" 이벤트를 수신함
```

**소스 위치**: `staging/src/k8s.io/client-go/tools/watch/`

### 7.4 플러그인 패턴 (Plugin Architecture)

스케줄러, 어드미션 컨트롤러 등에서 기능을 확장할 수 있는 구조임.

**소스 위치**: `plugin/`, `pkg/scheduler/framework/`
