# Kubernetes 코드 실행 흐름 완전 분석

> 이 문서는 "Kubernetes를 실행하면 코드가 실제로 어떻게 돌아가는지"를
> Go 소스 코드 수준에서 한 줄 한 줄 추적한 문서임.
> 추상적인 다이어그램이 아니라, **main() 함수부터 서버가 요청을 받기 시작하는 순간까지**를 추적함.

---

## 0. 먼저 알아야 할 것: 모든 컴포넌트의 공통 패턴

Kubernetes의 7개 바이너리(apiserver, kubelet, controller-manager, scheduler, proxy, kubectl, kubeadm)는
거의 동일한 구조로 시작함:

```go
// 거의 모든 컴포넌트의 main() 함수가 이렇게 생겼음
func main() {
    command := app.NewXXXCommand()    // cobra 명령 객체를 만듦
    code := cli.Run(command)          // 명령을 실행함
    os.Exit(code)                     // 종료 코드로 프로세스를 끝냄
}
```

**cobra란?** Go에서 가장 많이 쓰이는 CLI 프레임워크임. `kubectl get pods` 같은 명령줄을 파싱하고,
해당하는 함수를 호출해주는 라이브러리임. Docker, Hugo, GitHub CLI도 cobra를 사용함.

## 1. kube-apiserver: "모든 것의 중심"이 시작되는 과정

### 1단계: main() 함수 실행

**파일**: `cmd/kube-apiserver/apiserver.go`

```go
func main() {
    command := app.NewAPIServerCommand()   // API 서버 명령 객체를 만듦
    code := cli.Run(command)               // 실행함
    os.Exit(code)
}
```

여기서 `app.NewAPIServerCommand()`를 호출하면 다음 파일로 넘어감.

### 2단계: cobra 명령이 실행됨

**파일**: `cmd/kube-apiserver/app/server.go` (93-117행)

cobra의 `RunE` 콜백이 호출됨. 이 안에서:

```
① verflag.PrintAndExitIfRequested()     ← --version 플래그 처리
② s.Logs 설정 검증                       ← 로깅 시스템 초기화
③ completedOptions = s.Complete(ctx)     ← 설정값 완성 (기본값 채우기)
④ completedOptions.Validate()            ← 설정값 유효성 검증
⑤ Run(ctx, completedOptions)            ← 진짜 서버 시작 함수 호출
```

### 3단계: Run() — 서버 체인 생성

**파일**: `cmd/kube-apiserver/app/server.go` (148-172행)

```
⑤-a  config = NewConfig(opts)          ← 설정 객체 생성
⑤-b  config.Complete()                 ← 설정 완성
⑤-c  server = CreateServerChain(config) ← ★ 핵심: 3개의 서버를 체인으로 연결
```

**CreateServerChain이 하는 일** (175-196행):

```
apiExtensionsServer 생성   ← CRD(사용자 정의 리소스)를 처리하는 서버
        ↑ 위임(delegate)
kubeAPIServer 생성         ← Pod, Service 등 핵심 리소스를 처리하는 서버
        ↑ 위임(delegate)
aggregatorServer 생성      ← 외부 API 서버를 통합하는 서버 (최종 진입점)
```

**왜 3개인가?** 하나의 HTTP 요청이 들어오면:
1. aggregatorServer가 먼저 받음 → "이거 내가 처리할 수 있나?" 확인
2. 못 하면 kubeAPIServer에 위임 → "이거 Pod나 Service 요청인가?" 확인
3. 못 하면 apiExtensionsServer에 위임 → "이거 CRD 요청인가?" 확인

### 4단계: PrepareRun() — 엔드포인트 설치

**파일**: `cmd/kube-apiserver/app/server.go` (443-481행)

```
⑤-d  prepared = server.PrepareRun()
     ├── OpenAPI v2, v3 명세 엔드포인트 설치  ← /openapi/v2, /openapi/v3
     ├── /healthz 핸들러 설치                ← 헬스체크 엔드포인트
     ├── /livez 핸들러 설치                  ← 생존 확인 엔드포인트
     ├── /readyz 핸들러 설치                 ← 준비 상태 엔드포인트
     ├── /statusz 디버그 페이지 설치
     └── /flagz 디버그 페이지 설치
```

### 5단계: Run() — 드디어 서버가 요청을 받기 시작함

**파일**: `cmd/kube-apiserver/app/server.go` (536-638행)

```
⑤-e  prepared.Run(ctx)
     │
     ├── go UnprotectedDebugSocket.Run()    ← 디버그 소켓 고루틴 시작
     ├── go MuxAndDiscoveryComplete 감시     ← 모든 엔드포인트 등록 대기 고루틴
     ├── go shutdown 시그널 감시             ← SIGTERM 같은 종료 시그널 대기 고루틴
     │
     └── NonBlockingRunWithContext()         (762-790행)
         │
         ├── ★★★ SecureServingInfo.Serve(handler, timeout, stopCh) ★★★
         │   │
         │   └── 여기서 Go의 net/http.Server가 실제로 포트 6443에 바인딩하고
         │       TLS 연결을 받기 시작함
         │       → 이 순간부터 kubectl, kubelet 등이 API 서버에 접속할 수 있음
         │
         ├── RunPostStartHooks()            ← 서버 시작 후 실행할 훅들 실행
         └── systemd READY 알림 전송         ← systemd에 "나 준비됐어" 알림
```

### 요약: apiserver가 요청을 받을 준비가 되기까지

```
main()
 → app.NewAPIServerCommand()           [cobra 명령 생성]
   → RunE 콜백 실행                     [설정 검증]
     → Run()                            [서버 시작]
       → CreateServerChain()            [3개 서버 체인 연결]
         → PrepareRun()                 [엔드포인트 설치]
           → prepared.Run()             [고루틴 시작]
             → SecureServingInfo.Serve() [★ 포트 6443 리슨 시작!]
```

**소요 시간**: 보통 수 초 이내에 완료됨.
**이 시점 이후**: kubectl, kubelet, scheduler, controller-manager가 접속 가능해짐.

## 2. kubelet: "노드의 일꾼"이 시작되는 과정

### 1단계: main() 함수

**파일**: `cmd/kubelet/kubelet.go`

```go
func main() {
    ctx := context.Background()
    command := app.NewKubeletCommand(ctx)   // 다른 컴포넌트와 달리 ctx를 전달함
    code := cli.Run(command)
    os.Exit(code)
}
```

### 2단계: cobra 명령 → Run()

**파일**: `cmd/kubelet/app/server.go` (179-305행 → 539-552행)

```
RunE 콜백 실행
 ├── 플래그 파싱 및 검증
 ├── 설정 파일 로드 + CLI 플래그 병합
 ├── 로깅 초기화
 ├── Feature Gate 초기화
 └── Run(ctx, kubeletServer, kubeletDeps, featureGate)
```

### 3단계: run() — 진짜 초기화

**파일**: `cmd/kubelet/app/server.go` (621-1220행)

```
run()
 ├── NewHealthChecker()                  ← 워치독 초기화
 ├── CheckPermissions()                  ← 루트 권한 확인 (Linux)
 ├── 락 파일 획득                         ← 중복 실행 방지
 ├── GetHostname()                       ← 호스트명 결정
 ├── NodeName 결정                       ← API 서버에 등록할 노드 이름
 └── RunKubelet(ctx, ...)               ← 핵심 함수
```

### 4단계: RunKubelet() → createAndInitKubelet()

**파일**: `cmd/kubelet/app/server.go` (1223-1338행)

```
RunKubelet()
 ├── createAndInitKubelet()
 │   ├── kubelet.NewMainKubelet(...)     ← ★ kubelet 핵심 객체 생성
 │   │   이 함수 안에서:
 │   │   - 컨테이너 런타임(containerd)에 gRPC 연결
 │   │   - 볼륨 매니저 초기화
 │   │   - 이미지 매니저 초기화
 │   │   - 프로브(헬스체크) 매니저 초기화
 │   │   - cgroups 매니저 초기화
 │   │   - Pod 캐시 초기화
 │   │
 │   ├── k.BirthCry()                    ← API 서버에 "나 시작했어" 이벤트 전송
 │   └── k.StartGarbageCollection()      ← 죽은 컨테이너/이미지 정리 시작
 │
 └── startKubelet()                      ← 고루틴들 시작
```

### 5단계: startKubelet() — 고루틴들이 시작됨

**파일**: `cmd/kubelet/app/server.go` (1279-1291행)

```
startKubelet()
 │
 ├── go k.Run(podCfg.Updates())
 │   └── ★★★ kubelet 메인 루프 ★★★
 │       무한루프로 돌면서:
 │       - API 서버에서 "이 노드에서 실행해야 할 파드 목록"을 받아옴
 │       - 현재 실행 중인 파드와 비교함
 │       - 새로 추가된 파드 → containerd에 컨테이너 생성 지시
 │       - 삭제된 파드 → containerd에 컨테이너 중지 지시
 │       - 변경된 파드 → 업데이트 적용
 │       이 루프가 5-10초마다 반복됨
 │
 ├── go k.ListenAndServe()
 │   └── ★ 포트 10250에서 HTTPS 서버 시작 ★
 │       kubectl exec, kubectl logs 같은 명령이 이 포트로 접속함
 │
 ├── go k.ListenAndServeReadOnly()
 │   └── 포트 10255에서 읽기 전용 HTTP 서버 시작 (메트릭 등)
 │
 └── go k.ListenAndServePodResources()
     └── 디바이스 플러그인용 gRPC 엔드포인트 시작
```

### 요약: kubelet이 파드를 실행할 준비가 되기까지

```
main()
 → NewKubeletCommand(ctx)
   → RunE 실행
     → Run()
       → run()
         → RunKubelet()
           → createAndInitKubelet()
             → NewMainKubelet()       [containerd 연결, 매니저들 초기화]
             → BirthCry()             [apiserver에 "시작" 이벤트]
           → startKubelet()
             → go k.Run()            [★ 메인 파드 동기화 루프 시작]
             → go ListenAndServe()   [★ 포트 10250 리슨 시작]
```

**이 시점 이후**: 스케줄러가 이 노드에 파드를 배치하면, kubelet이 감지하고 containerd에 컨테이너 생성을 지시함.

## 3. kube-controller-manager: "38개의 감시자"가 시작되는 과정

### 1단계~2단계: main() → cobra

**파일**: `cmd/kube-controller-manager/controller-manager.go`

```go
func main() {
    command := app.NewControllerManagerCommand()
    code := cli.Run(command)
    os.Exit(code)
}
```

### 3단계: Run() — 핵심 로직

**파일**: `cmd/kube-controller-manager/app/controllermanager.go` (185-427행)

```
Run()
 │
 ├── EventBroadcaster.StartStructuredLogging()
 │   └── 이벤트를 구조화된 로그로 출력하기 시작함
 │
 ├── EventBroadcaster.StartRecordingToSink()
 │   └── 이벤트를 API 서버에 기록하기 시작함
 │       (예: "Deployment nginx가 스케일업됨" 같은 이벤트)
 │
 ├── ★ 헬스체크/메트릭 HTTP 서버 시작 (217-241행) ★
 │   ├── NewBaseHandler() → 기본 핸들러 생성
 │   ├── /metrics, /healthz, /readyz 핸들러 설치
 │   └── SecureServing.Serve() → 포트 10257에서 리슨 시작
 │
 ├── createClientBuilders()
 │   └── API 서버에 접속할 클라이언트 빌더 생성
 │
 └── ★★★ 리더 선출 + 컨트롤러 시작 (280-427행) ★★★
```

### 4단계: 리더 선출 — "나만 일할래"

```
리더 선출이 왜 필요한가?
→ controller-manager를 고가용성(HA)을 위해 2-3개 복제본으로 실행할 수 있음
→ 하지만 Deployment 복제본 수를 동시에 2곳에서 조절하면 충돌함
→ 그래서 한 번에 하나의 인스턴스만 "리더"가 되어 일하고,
  나머지는 대기하다가 리더가 죽으면 새 리더가 됨

구현 방식:
→ kube-system 네임스페이스에 Lease 리소스를 만듦
→ 이 Lease를 가진 인스턴스가 리더임
→ 리더는 주기적으로 Lease를 갱신함 (보통 15초)
→ 갱신이 안 되면 다른 인스턴스가 Lease를 가져감
```

```
리더 선출 흐름:

IF 리더 선출 비활성화:
  → run(ctx, controllers) 직접 호출    ← 바로 컨트롤러 시작

ELSE:
  → go leaderElectAndRun()             ← 고루틴으로 리더 선출 시작
     │
     ├── Lease 리소스 획득 시도 (반복)
     │
     └── OnStartedLeading 콜백:        ← 리더가 되면 호출됨
         └── run(ctx, controllers)      ← 컨트롤러 시작!
```

### 5단계: run() → RunControllers() — 38개 컨트롤러 시작

```
run()
 ├── CreateControllerContext()          ← 공유 컨텍스트 생성
 ├── BuildControllers()                 ← 38개 컨트롤러 인스턴스 생성
 ├── InformerFactory.Start(stopCh)      ← ★ Informer 시작 (API 서버 Watch 시작)
 │   └── 이 순간 API 서버에 연결하여 리소스 변경을 감시하기 시작함
 │       예: "Pod가 생겼어", "Service가 변경됐어" 같은 이벤트를 실시간으로 수신
 │
 └── RunControllers()                   (655-750행)
     │
     FOR EACH controller (38개):
       go func() {                      ← 각 컨트롤러마다 고루틴 생성
         time.Sleep(랜덤 지연)           ← 동시 시작으로 인한 부하 분산
         controller.Run(ctx)            ← ★ 컨트롤러 메인 루프 시작
       }()
```

### 각 컨트롤러의 Run()이 하는 일 (Reconciliation Loop)

```
controller.Run(ctx)
 └── 무한루프:
     ①  작업 큐(WorkQueue)에서 항목 하나를 가져옴
         (예: "default/nginx Deployment가 변경됨")

     ②  현재 상태를 확인함
         (예: "nginx ReplicaSet의 파드가 2개 실행 중")

     ③  원하는 상태와 비교함
         (예: "Deployment에 replicas: 3이 지정되어 있으니 3개여야 함")

     ④  차이를 발견하면 조치함
         (예: "파드 1개를 더 생성해야 함" → API 서버에 Pod 생성 요청)

     ⑤  ①로 돌아감
```

**구체적인 예시 — Deployment 컨트롤러:**
```
사용자가 kubectl apply -f nginx.yaml (replicas: 3) 실행
                    │
                    ▼
API 서버가 Deployment 저장 → Watch 이벤트 발생
                    │
                    ▼
Informer가 이벤트를 수신 → 작업 큐에 "default/nginx" 추가
                    │
                    ▼
Deployment 컨트롤러가 큐에서 꺼냄
                    │
                    ▼
"ReplicaSet이 없네?" → API 서버에 ReplicaSet 생성 요청
                    │
                    ▼
ReplicaSet 컨트롤러가 Watch로 감지
                    │
                    ▼
"파드가 0개인데 3개여야 하네?" → API 서버에 Pod 3개 생성 요청
                    │
                    ▼
스케줄러가 Watch로 Pending 파드 감지 → 노드 배정
                    │
                    ▼
kubelet이 Watch로 자기 노드에 배정된 파드 감지 → containerd에 컨테이너 생성
```

## 4. kube-scheduler: "파드를 어디에 놓을지 결정하는 놈"

### 시작 과정 (Run 함수)

**파일**: `cmd/kube-scheduler/app/server.go` (171-340행)

```
Run()
 │
 ├── EventBroadcaster 시작              ← 스케줄링 이벤트를 API 서버에 기록
 │
 ├── ★ HTTPS 서버 시작 (250-267행) ★
 │   └── 포트 10259에서 리슨
 │
 ├── startInformersAndWaitForSync()     ← ★ API 서버 Watch 시작
 │   ├── InformerFactory.Start()         ← Pod, Node, PV 등 감시 시작
 │   ├── DynInformerFactory.Start()      ← 동적 리소스 감시 시작
 │   ├── WaitForCacheSync()              ← 로컬 캐시가 최신 상태가 될 때까지 대기
 │   │   (이 단계가 끝나야 스케줄링을 시작할 수 있음 —
 │   │    캐시가 비어있으면 노드 정보를 모르니까 배치를 못 함)
 │   └── WaitForHandlersSync()           ← 스케줄러 핸들러 동기화 대기
 │
 └── ★★★ sched.Run(ctx) — 스케줄링 메인 루프 ★★★
```

### sched.Run()이 하는 일

```
sched.Run(ctx)
 └── 무한루프:
     ①  스케줄링 큐에서 Pending 상태의 파드를 꺼냄

     ②  Pre-Filter 플러그인 실행
         (예: "이 파드는 GPU가 필요함" 같은 사전 조건 확인)

     ③  Filter (필터링): 모든 노드를 검사
         "이 노드에 리소스가 충분한가?"
         "이 노드의 taint가 파드의 toleration과 맞는가?"
         "이 파드의 nodeSelector와 노드 라벨이 일치하는가?"
         → 부적합한 노드를 제거함

     ④  Score (점수 매기기): 남은 노드들에 점수 부여
         "리소스가 많이 남은 노드에 높은 점수"
         "같은 서비스의 파드가 이미 있는 노드에 낮은 점수 (분산 배치)"
         → 가장 높은 점수의 노드를 선택함

     ⑤  Reserve: 선택된 노드의 리소스를 예약함

     ⑥  Bind: API 서버에 "이 파드를 이 노드에 배치해줘" 요청
         → Binding 객체를 생성하여 API 서버에 전송함

     ⑦  Post-Bind: 후처리 (로깅, 메트릭 등)

     ⑧  ①로 돌아감
```

## 5. kube-proxy: "네트워크 규칙을 관리하는 놈"

### 시작 과정

**파일**: `cmd/kube-proxy/app/server.go`

```
RunE → opts.Run()
 │
 ├── newProxyServer()                    ← 프록시 서버 객체 생성
 │   ├── createClient()                  ← API 서버 클라이언트 생성
 │   │   ├── rest.InClusterConfig()      ← 파드 안에서 실행 중이면 자동 설정
 │   │   └── clientset.NewForConfig()    ← ★ API 서버에 연결
 │   ├── NewNodeManager()                ← 자기 노드 정보 감시
 │   └── createProxier()                 ← ★ iptables/ipvs/nftables 프록시어 생성
 │       (OS와 설정에 따라 3가지 중 하나를 선택함)
 │
 └── runLoop()
     │
     └── go proxyServer.Run()
         │
         ├── 헬스체크 서버 시작           ← 포트 10256
         ├── 메트릭 서버 시작             ← 포트 10249
         │
         ├── SharedInformerFactory 생성
         ├── EndpointSliceConfig 생성 + 이벤트 핸들러 등록
         ├── ServiceConfig 생성 + 이벤트 핸들러 등록
         ├── informerFactory.Start()     ← ★ API 서버 Watch 시작
         │   └── Service와 EndpointSlice 리소스 변경을 감시함
         │
         └── ★★★ go Proxier.SyncLoop() — 메인 루프 ★★★
```

### Proxier.SyncLoop()이 하는 일

```
SyncLoop()
 └── 무한루프:
     ①  Service 또는 EndpointSlice 변경 이벤트를 기다림

     ②  변경이 감지되면:
         IF iptables 모드:
           → iptables 규칙을 재생성함
           예: iptables -t nat -A KUBE-SERVICES -d 10.96.0.100/32 -p tcp --dport 80
               -j KUBE-SVC-XXXXX
           → Service IP(10.96.0.100)로 오는 트래픽을 실제 파드 IP들로 분산시키는 규칙

         IF ipvs 모드:
           → IPVS 가상 서버와 실제 서버 규칙을 업데이트함

         IF nftables 모드:
           → nftables 규칙을 업데이트함

     ③  ①로 돌아감
```

**구체적 예시:**
```
사용자가 Service 생성: type=ClusterIP, port=80, selector=app:nginx

kube-proxy가 Watch로 감지:
  "새 서비스 10.96.0.100:80 → 파드 10.244.1.5:8080, 10.244.2.3:8080"

iptables 규칙 생성:
  -A KUBE-SERVICES -d 10.96.0.100 -p tcp --dport 80 → KUBE-SVC-NGINX
  -A KUBE-SVC-NGINX → 50% 확률로 10.244.1.5:8080 (DNAT)
  -A KUBE-SVC-NGINX → 50% 확률로 10.244.2.3:8080 (DNAT)

결과: 클러스터 안에서 10.96.0.100:80으로 요청하면
      실제 nginx 파드 중 하나로 트래픽이 전달됨
```

## 7. 전체 그림: 모든 컴포넌트가 함께 동작하는 모습

```
[시작 순서]

1. etcd 시작                    ← 데이터 저장소 먼저 준비
   └── 포트 2379에서 리슨

2. kube-apiserver 시작          ← etcd에 연결
   └── 포트 6443에서 리슨
   └── SecureServingInfo.Serve()가 호출된 시점

3. kube-controller-manager 시작 ← apiserver에 연결
   └── InformerFactory.Start() → apiserver Watch 시작
   └── 리더 선출 → RunControllers() → 38개 고루틴 시작

4. kube-scheduler 시작          ← apiserver에 연결
   └── InformerFactory.Start() → apiserver Watch 시작
   └── WaitForCacheSync() → 캐시 동기화 완료 대기
   └── sched.Run() → 스케줄링 루프 시작

5. kubelet 시작 (각 워커 노드)  ← apiserver에 연결, containerd에 연결
   └── BirthCry() → "나 시작했어" 이벤트
   └── go k.Run() → 파드 동기화 루프 시작
   └── go ListenAndServe() → 포트 10250 리슨

6. kube-proxy 시작 (각 워커 노드) ← apiserver에 연결
   └── informerFactory.Start() → Service/EndpointSlice Watch 시작
   └── go Proxier.SyncLoop() → iptables 규칙 동기화 루프 시작
```

```
[파드 하나가 생성되는 전체 여정]

사용자: kubectl apply -f nginx.yaml
  │
  │ HTTPS POST → apiserver:6443
  ▼
apiserver: Pod 객체를 etcd에 저장 (status: Pending)
  │
  │ Watch 이벤트 발생
  ▼
scheduler: "Pending 파드 발견!"
  → Filter: 노드 3개 중 리소스 충분한 2개 선택
  → Score: worker-2가 점수 높음
  → Bind: apiserver에 "이 파드를 worker-2에 배치" 요청
  │
  │ Watch 이벤트 발생
  ▼
kubelet (worker-2): "내 노드에 새 파드 발견!"
  → containerd에 gRPC 호출: "nginx:latest 이미지 Pull해"
  → containerd에 gRPC 호출: "컨테이너 만들어"
  → containerd에 gRPC 호출: "컨테이너 시작해"
  → apiserver에 상태 보고: "파드 Running"
  │
  │ Watch 이벤트 발생
  ▼
kube-proxy (모든 노드): "이 파드가 Service에 연결되어 있네"
  → iptables 규칙 업데이트: 새 파드 IP를 Service 엔드포인트에 추가
```

## 8. 핵심 키워드 정리

| 코드에서 보이는 것 | 실제로 하는 일 |
|-------------------|---------------|
| `cobra.Command.RunE` | 프로그램이 실행될 때 호출되는 메인 함수임 |
| `cli.Run(command)` | cobra 명령을 실행하고 종료 코드를 반환함 |
| `SecureServingInfo.Serve()` | 실제로 HTTPS 포트를 열고 요청을 받기 시작하는 순간임 |
| `InformerFactory.Start()` | API 서버에 Watch 연결을 만들어 리소스 변경을 실시간 감시하기 시작함 |
| `WaitForCacheSync()` | 로컬 캐시가 API 서버와 동기화될 때까지 대기함 (이게 끝나야 일을 시작할 수 있음) |
| `go func() { controller.Run() }` | 새 고루틴(경량 스레드)을 만들어서 컨트롤러 루프를 돌림 |
| `leaderElectAndRun()` | 여러 인스턴스 중 하나만 일하도록 리더를 선출함 |
| `Proxier.SyncLoop()` | iptables/ipvs 규칙을 지속적으로 동기화하는 무한루프임 |
| `k.Run(podCfg.Updates())` | kubelet이 파드 변경사항을 감시하며 컨테이너를 관리하는 메인루프임 |
| `rest.InClusterConfig()` | 파드 안에서 실행 중일 때 자동으로 API 서버 연결 정보를 가져옴 |
