> 가장 높은 수준에서 Kubernetes 시스템과 외부 엔터티 간의 데이터 흐름을 보여주는 다이어그램임.

### 핸들러 체인 조립 순서 (config.go 1028행)

`DefaultBuildHandlerChain()`은 안쪽부터 바깥쪽으로 래핑하는 방식임. 요청이 처리되는 실제 순서(바깥→안)는 다음과 같음:

```
요청 → HTTPLogging → RequestDeadline → Timeout → CORS
     → Authentication (1069행)
     → Audit (1056행)
     → Impersonation (1051행)
     → Authorization (1032행)
     → PriorityAndFairness (1040행)
     → API 핸들러
```

## 4. Watch/Informer 메커니즘 상세 분석

> Kubernetes의 **실시간 이벤트 스트림** 시스템을 분석한 것임. 이것이 Kubernetes의 핵심 통신 방식임.

### 4.1 Watch의 동작 원리

```
┌───────────────┐              ┌──────────────┐
│ 클라이언트    │   HTTP GET   │              │
│ (scheduler,   │ ─────────→  │ kube-api     │
│  controller,  │ ?watch=true  │ server       │
│  kubelet)     │              │              │
│               │ ←─────────── │              │
│               │ chunked      │              │
│               │ transfer     │              │
│               │ (이벤트      │              │
│               │  스트림)     │              │
└───────────────┘              └──────────────┘

[이벤트 스트림 예시]
───────────────────────────────────────
{"type":"ADDED",   "object":{"kind":"Pod","metadata":{"name":"nginx-1"}...}}
{"type":"MODIFIED","object":{"kind":"Pod","metadata":{"name":"nginx-1"}...}}
{"type":"DELETED", "object":{"kind":"Pod","metadata":{"name":"nginx-1"}...}}
───────────────────────────────────────
```

### 4.2 Informer 패턴 (상세 소스 분석)

## 10. 초보자를 위한 핵심 요약

### "데이터는 이렇게 흐름"

```
[간략화된 전체 흐름]

사용자 ──(kubectl)──→ API서버 ──(저장)──→ etcd
                        │
                   (Watch 알림)
                   ┌────┼────┐
                   ▼    ▼    ▼
              스케줄러 컨트롤러 kubelet
                              │
                         (CRI/gRPC)
                              ▼
                         containerd
                              │
                         컨테이너 실행
```

**핵심 포인트**:
1. **모든 통신은 API 서버를 중심으로** 이루어짐 (Hub-and-Spoke 패턴)
2. **etcd는 유일한 상태 저장소**임 — API 서버만 etcd에 직접 접근함 (`storage.Interface` 169행)
3. **Watch 메커니즘**이 실시간 이벤트 전달의 핵심임 (`watcher.go` — etcd gRPC → `resultChan` → 구독자)
4. **Informer 패턴**이 Watch를 효율적으로 관리함 (`SharedInformer` 143행 — 캐싱 + 이벤트 핸들러)
5. **컨트롤러들은 API 서버를 통해** 간접적으로 상태를 변경함 (직접 etcd 접근하지 않음)
