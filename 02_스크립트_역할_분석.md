# Kubernetes 스크립트 역할 분석서

> 소프트웨어 공학의 **기능 분해(Functional Decomposition)** 기법을 활용하여,
> Kubernetes 프로젝트의 스크립트들을 실제 소스 코드 수준에서 분석한 문서임.
> 각 스크립트의 **행 수, 함수명, 변수명, 호출 관계, 실행 흐름**을 구체적으로 기술함.

---

## 1. 스크립트 구조 전체 지도

Kubernetes 프로젝트에는 약 **308개의 쉘 스크립트**가 존재하며, 크게 3개의 디렉토리에 분포함.

```
kubernetes/
├── build/     ← 8개 핵심 스크립트 + lib/ (빌드/릴리즈)
├── hack/      ← 200개 이상 (개발자용 도구, 가장 많음)
│   ├── lib/           ← 5개 라이브러리 스크립트
│   ├── make-rules/    ← 7개 Make 타겟 구현체
│   ├── verify-*.sh    ← 50개 이상 검증 스크립트
│   └── update-*.sh    ← 30개 이상 코드 생성 스크립트
└── cluster/   ← 7개 클러스터 배포 스크립트 (유지보수 모드)
```

## 2. build/ 디렉토리 — 빌드 및 릴리즈 스크립트

### 2.1 build/common.sh (473행, 19개 함수)

> **역할**: 모든 빌드 스크립트가 `source`하는 **공통 라이브러리**임. Docker 컨테이너 관리, 이미지 설정, 경로 관리 함수를 제공함.

**소스 시작 시 수행하는 작업:**
```bash
set -o errexit    # 에러 발생 시 즉시 종료함
set -o nounset    # 미정의 변수 사용 시 에러를 발생시킴
set -o pipefail   # 파이프라인 중간 실패도 감지함
```

**의존성:** `hack/lib/init.sh`를 source함

**정의된 핵심 변수:**

| 변수명 | 설정값 | 역할 |
|--------|--------|------|
| `KUBE_ROOT` | 프로젝트 루트 경로 | 모든 스크립트가 이 변수를 기준으로 경로를 계산함 |
| `KUBE_DOCKER_REGISTRY` | `registry.k8s.io` | Docker 이미지가 푸시되는 기본 레지스트리임 |
| `KUBE_BASE_IMAGE_REGISTRY` | `registry.k8s.io/build-image` | 빌드 기본 이미지들의 레지스트리임 |
| `KUBE_CROSS_IMAGE` | `registry.k8s.io/build-image/kube-cross` | 크로스 컴파일용 Docker 이미지임 |
| `LOCAL_OUTPUT_ROOT` | `${KUBE_ROOT}/_output` | 빌드 산출물이 저장되는 로컬 루트 경로임 |
| `LOCAL_OUTPUT_BINPATH` | `${LOCAL_OUTPUT_ROOT}/dockerized/bin` | 컴파일된 바이너리가 저장되는 경로임 |
| `REMOTE_ROOT` | `/go/src/k8s.io/kubernetes` | Docker 컨테이너 안에서의 소스 코드 경로임 |
| `KUBE_BUILD_SETCAP_IMAGE` | setcap 이미지 | kube-apiserver에 커널 캐퍼빌리티를 설정하기 위한 이미지임 |

**컴포넌트별 기본 이미지 매핑:**

| 변수명 | 이미지 | 용도 |
|--------|--------|------|
| `KUBE_APISERVER_BASE_IMAGE` | `distroless-iptables` | apiserver는 동적 링킹 바이너리이므로 iptables 포함 이미지를 사용함 |
| `KUBE_CONTROLLER_MANAGER_BASE_IMAGE` | `go-runner` | 정적 링킹 바이너리이므로 경량 이미지를 사용함 |
| `KUBE_SCHEDULER_BASE_IMAGE` | `go-runner` | 정적 링킹 바이너리임 |
| `KUBE_PROXY_BASE_IMAGE` | `distroless-iptables` | iptables가 필요하므로 iptables 포함 이미지를 사용함 |
| `KUBECTL_BASE_IMAGE` | `go-runner` | 정적 링킹 바이너리임 |

### 2.2 build/run.sh (37행)

> **역할**: Docker 빌드 컨테이너 안에서 사용자가 지정한 명령을 실행하는 래퍼 스크립트임.

**실행 흐름:**
```
1. set -o errexit/nounset/pipefail        ← 에러 핸들링 설정
2. KUBE_ROOT 계산                          ← 스크립트 위치 기반
3. source build/common.sh                  ← 공통 라이브러리 로드
4. KUBE_RUN_COPY_OUTPUT=${KUBE_RUN_COPY_OUTPUT:-y}  ← 비활성화 경고 (deprecated)
5. kube::build::verify_prereqs()           ← Docker, tar 등 전제조건 확인
6. kube::build::run_build_command "$@"     ← 전달받은 명령을 Docker 안에서 실행
```

**사용 예시:**
```bash
build/run.sh make                    # Docker 안에서 make 실행
build/run.sh make test WHAT=./pkg/api  # Docker 안에서 특정 패키지 테스트
```

### 2.3 build/shell.sh (29행)

> **역할**: Docker 빌드 컨테이너에 대화형 Bash 쉘로 접속하는 기능을 함.

**실행 흐름:**
```
1. source build/common.sh
2. KUBE_RUN_COPY_OUTPUT='n'               ← 쉘 종료 시 출력 복사 비활성화
3. ${KUBE_ROOT}/build/run.sh bash "$@"    ← run.sh를 통해 bash를 Docker 안에서 실행
```

**용도**: 빌드 환경 내부를 디버깅하거나 수동으로 빌드 명령을 실험할 때 사용함.

### 2.4 build/release.sh (43행)

> **역할**: **전체 릴리즈 프로세스**(빌드 → 테스트 → 패키징)를 순서대로 실행하는 오케스트레이터 스크립트임.

**실행 흐름:**
```
1. source build/common.sh                            ← 공통 라이브러리
2. source build/lib/release.sh                       ← 릴리즈 전용 함수 (package_tarballs 등)
3. KUBE_RELEASE_RUN_TESTS=${KUBE_RELEASE_RUN_TESTS:-y}  ← 테스트 실행 여부 (기본: 실행)
4. kube::build::verify_prereqs()                     ← 전제조건 확인
5. kube::build::run_build_command make cross          ← [빌드] 모든 플랫폼용 크로스 컴파일
6. if KUBE_RELEASE_RUN_TESTS == y:
   6a. kube::build::run_build_command make test        ← [테스트] 유닛 테스트 실행
   6b. kube::build::run_build_command make test-integration  ← [통합테스트] 통합 테스트 실행
7. kube::release::package_tarballs()                  ← [패키징] tar.gz 아카이브 생성
```

**분기 조건**: `KUBE_RELEASE_RUN_TESTS=n`으로 설정하면 6a, 6b 단계를 건너뜀 (`make quick-release`가 이 방식을 사용함).

### 2.5 build/release-images.sh (41행)

> **역할**: 서버 컴포넌트(apiserver, controller-manager, scheduler, proxy, kubectl)의 **Docker 이미지를 생성**하는 기능을 함.

**실행 흐름:**
```
1. source build/common.sh + build/lib/release.sh
2. CMD_TARGETS = KUBE_SERVER_IMAGE_TARGETS                ← 서버 이미지 빌드 대상
3. if KUBE_BUILD_CONFORMANCE == y:
     CMD_TARGETS += KUBE_CONFORMANCE_IMAGE_TARGETS        ← 적합성 테스트 이미지 추가
4. if KUBE_EXTRA_WHAT != "":
     CMD_TARGETS += KUBE_EXTRA_WHAT                        ← 추가 빌드 타겟
5. kube::build::verify_prereqs()
6. kube::build::run_build_command make all WHAT="${CMD_TARGETS}" ...  ← 바이너리 빌드
7. kube::release::build_server_images()                    ← Docker 이미지 생성
```

`kube::release::build_server_images()` 내부에서:
- `docker buildx build --platform linux/${arch}`로 멀티 아키텍처 이미지를 빌드함
- 각 바이너리마다 `build/server-image/{바이너리명}/Dockerfile`이 있으면 해당 Dockerfile을, 없으면 공용 `build/server-image/Dockerfile`을 사용함

### 2.7 build/package-tarballs.sh (28행)

> **역할**: 빌드된 바이너리를 플랫폼별 tar.gz 아카이브로 패키징하는 기능을 함.

**실행 흐름:**
```
1. source build/common.sh + build/lib/release.sh
2. kube::build::ensure_tar()                  ← GNU tar 사용 가능 확인
3. kube::version::get_version_vars()          ← git tag 기반 버전 정보 추출
4. kube::release::package_tarballs()          ← 실제 패키징 실행
```

`kube::release::package_tarballs()` 함수가 생성하는 파일들:
```
_output/release-tars/
├── kubernetes-src.tar.gz                    ← 소스 코드 아카이브
├── kubernetes-client-linux-amd64.tar.gz     ← 플랫폼별 클라이언트 바이너리
├── kubernetes-client-darwin-arm64.tar.gz
├── kubernetes-node-linux-amd64.tar.gz       ← 플랫폼별 노드 바이너리
├── kubernetes-server-linux-amd64.tar.gz     ← 플랫폼별 서버 바이너리 + Docker 이미지
└── kubernetes-manifests.tar.gz              ← kube-system 매니페스트
```

### 2.9 build/lib/release.sh (400행+, 12개 함수)

> **역할**: 릴리즈 패키징과 Docker 이미지 생성의 **실제 구현체**임. `build/release.sh`, `build/release-images.sh`, `build/package-tarballs.sh`가 이 파일의 함수들을 호출함.

**핵심 함수:**

| 함수명 | 구체적 역할 |
|--------|-------------|
| `kube::release::package_tarballs()` | 메인 패키징 오케스트레이터임. src, client, node, server, test, manifests 타볼을 **병렬로** 생성함 |
| `kube::release::package_src_tarball()` | `git archive`로 소스 코드 아카이브를 생성함. dirty tree이면 `find/tar` 폴백을 사용함 |
| `kube::release::package_client_tarballs()` | `KUBE_BUILD_PLATFORMS` 배열을 순회하며 플랫폼별 클라이언트(kubectl 등) tar.gz를 생성함. Windows는 `.exe` 확장자를 처리함 |
| `kube::release::package_node_tarballs()` | 플랫폼별 노드 바이너리(kubelet, kube-proxy) + 클라이언트 바이너리를 합쳐서 패키징함 |
| `kube::release::build_server_images()` | `docker buildx build --platform` 명령으로 서버 컴포넌트의 Docker 이미지를 빌드함 |
| `kube::release::package_server_tarballs()` | `build_server_images()`를 호출한 후 서버 바이너리 + 이미지를 합쳐서 패키징함 |
| `kube::release::create_docker_images_for_server()` | 바이너리별로 적절한 Dockerfile을 선택하고, BASEIMAGE/SETCAP_IMAGE/BINARY 빌드 인자를 전달하여 이미지를 빌드함 |
| `kube::release::build_conformance_image()` | CNCF 적합성 테스트용 Docker 이미지를 빌드함 |
| `kube::release::package_kube_manifests_tarball()` | `cluster/gce/manifests/` 에서 kube-system 매니페스트(kube-proxy, etcd, scheduler, apiserver)를 패키징함 |

## 3. hack/ 디렉토리 — 개발자 도구 스크립트

### 3.1 hack/lib/init.sh (219행) — 모든 스크립트의 출발점

> **역할**: Kubernetes 스크립트 생태계의 **초기화 진입점**임. 거의 모든 스크립트가 이 파일을 source함.

**실행 흐름:**
```
1. set -o errexit/nounset/pipefail           ← 엄격한 에러 처리 활성화
2. unset CDPATH                               ← cd 명령의 부작용 방지
3. KUBE_ROOT 계산                             ← 스크립트 위치에서 프로젝트 루트를 역추적
4. KUBE_OUTPUT="${KUBE_ROOT}/_output/local"   ← 빌드 출력 디렉토리 설정
5. source hack/lib/util.sh                    ← 범용 유틸리티 함수 로드
6. source hack/lib/logging.sh                 ← 로깅 함수 로드
7. source hack/lib/version.sh                 ← 버전 정보 함수 로드
8. source hack/lib/golang.sh                  ← Go 빌드 환경 함수 로드
9. source hack/lib/etcd.sh                    ← etcd 관리 함수 로드
10. kube::log::install_errexit                ← ERR 트랩 핸들러 설치
11. kube::util::ensure-bash-version           ← Bash 버전 확인
```

**핵심 변수 (61-131행):**
`KUBE_AVAILABLE_GROUP_VERSIONS` — Kubernetes가 지원하는 **모든 API 그룹/버전** 목록이 정의되어 있음:
```
v1 admissionregistration.k8s.io/v1 apps/v1 autoscaling/v1 autoscaling/v2
batch/v1 certificates.k8s.io/v1 coordination.k8s.io/v1 discovery.k8s.io/v1
events.k8s.io/v1 networking.k8s.io/v1 node.k8s.io/v1 policy/v1
rbac.authorization.k8s.io/v1 scheduling.k8s.io/v1 storage.k8s.io/v1
... (Alpha/Beta 포함 50개 이상)
```
