# Kubernetes 의존성 분석서

> **사용 기법**: 소프트웨어 공학의 **의존성 구조 행렬(DSM, Dependency Structure Matrix)** 및 **패키지 의존 그래프(Dependency Graph)** 기법을 활용하여 Kubernetes 내부 모듈 간 의존 관계를 분석한 문서임.
> 모든 소스 위치는 `/home/c1_master1/kubernetes/` 기준이며, 실제 `go.mod`, `go.work` 파일에서 확인한 내용임.

### 의존성 관리 핵심 파일

| 파일 | 구체적 역할 |
|------|-------------|
| `go.mod` | 루트 모듈(`k8s.io/kubernetes`)의 직접/간접 의존성 정의. go 1.25.7 지정 |
| `go.sum` | 모든 의존성의 SHA-256 체크섬. 무결성 검증에 사용됨 |
| `go.work` | 루트(`.`) + 31개 `staging/src/k8s.io/*` 모듈을 `use` 지시문으로 등록 (총 32개 모듈). go 1.25.0 지정 |
| `vendor/modules.txt` | 벤더링된 모듈 목록과 메타데이터 |
| `hack/update-vendor.sh` | 347행. 의존성 동기화의 **마스터 스크립트**. 12단계로 go.mod→go.work→vendor를 관리함 |
| `hack/verify-vendor.sh` | vendor/ 디렉토리가 go.mod와 일치하는지 검증함 |
| `hack/verify-no-vendor-cycles.sh` | staging 모듈 간 순환 의존성이 없는지 검증함 |
| `hack/verify-import-boss.sh` | `.import-restrictions` 파일로 패키지 간 import 계층 규칙을 강제함 |
| `hack/verify-govulncheck.sh` | `govulncheck`로 의존성의 알려진 보안 취약점을 스캔함 |
| `hack/verify-licenses.sh` | 모든 의존성 라이선스가 Apache 2.0 호환인지 검증함 |

## 2. 계층별 의존성 구조

> Kubernetes의 코드는 명확한 **계층(Layer)** 구조를 가지며, 의존성은 상위 → 하위로만 허용됨.

```
┌──────────────────────────────────────────────────────┐
│ Layer 4 (최상위): cmd/ — 실행 파일 진입점            │
│   kube-apiserver, kubelet, kubectl, 등               │
│   → Layer 3, Layer 2, Layer 1 모두에 의존함          │
├──────────────────────────────────────────────────────┤
│ Layer 3: pkg/ — Kubernetes 전용 비즈니스 로직        │
│   controller, scheduler, kubelet 구현체 등           │
│   → Layer 2, Layer 1에 의존함                        │
├──────────────────────────────────────────────────────┤
│ Layer 2: staging/ — 공유 라이브러리                   │
│   client-go, apimachinery, apiserver 등              │
│   → Layer 1에만 의존함                               │
├──────────────────────────────────────────────────────┤
│ Layer 1 (최하위): vendor/ — 외부 의존성              │
│   etcd, gRPC, Prometheus, cobra 등                   │
│   → 외부 세계에 의존함                               │
└──────────────────────────────────────────────────────┘

[허용]  상위 → 하위 (cmd → pkg → staging → vendor)
[금지]  하위 → 상위 (vendor → staging, staging → pkg 등)
[검증]  hack/verify-import-boss.sh 가 이 규칙을 강제함
```

## 4. staging/ 모듈 간 내부 의존성

> 31개 staging 모듈 간의 의존 관계를 분석한 것임. 화살표(→)는 "~가 ~에 의존함"을 의미함.

### 4.1 의존성 구조 행렬 (핵심 모듈)

```
              api  apimach  apisvr  client-go  comp-base  cli-rt  kubectl
api            -     ✓        -        -          -         -       -
apimachinery   -     -        -        -          -         -       -
apiserver      ✓     ✓        -        ✓          ✓         -       -
client-go      ✓     ✓        -        -          -         -       -
component-base -     ✓        -        ✓          -         -       -
cli-runtime    ✓     ✓        -        ✓          -         -       -
kubectl        ✓     ✓        -        ✓          ✓         ✓       -

✓ = 의존함, - = 의존하지 않음
```

**읽는 법**: 행(가로)이 "사용하는 모듈", 열(세로)이 "사용되는 모듈"임.
- 예: `client-go`는 `api`와 `apimachinery`에 의존함.
- 예: `kubectl`은 거의 모든 모듈에 의존함 (최상위 소비자).

### 4.2 의존성 계층 다이어그램

```
Level 0 (기반):
  ┌──────────────┐

  └──────┬───────┘   다른 모든 모듈이 이것에 의존함
         │
Level 1:
  ┌──────▼───────┐   ┌──────────────┐
```

## 6. 의존성 관리 도구 및 검증

### 6.1 의존성 관리 파일

| 파일 | 역할 |
|------|------|
| `go.mod` | 프로젝트의 직접 의존성을 정의하는 파일임 |
| `go.sum` | 의존성의 체크섬을 기록하여 무결성을 보장하는 파일임 |
| `go.work` | 32개 모듈을 하나의 워크스페이스로 관리하는 파일임. `go 1.25.0` 지정 |
| `go.work.sum` | 워크스페이스 의존성 체크섬 파일임 |
| `vendor/` | 모든 의존성의 소스 코드를 로컬에 복사한 디렉토리임 |
| `vendor/modules.txt` | 벤더링된 모듈 목록 및 메타데이터 파일임 |

### 6.2 의존성 검증 스크립트

| 스크립트 | 역할 |
|----------|------|
| `hack/verify-vendor.sh` | vendor/ 디렉토리가 go.mod와 일치하는지 검증하는 기능을 함 |

## 7. 순환 의존성 방지 메커니즘

> Kubernetes 프로젝트는 순환 의존성을 매우 엄격하게 금지함.

### 7.1 import-boss 규칙

```
[허용되는 의존 방향]
  cmd/ ──→ pkg/ ──→ staging/ ──→ vendor/
  (상위)              (하위)

[금지되는 의존 방향]
  staging/ ──✗──→ pkg/       (하위가 상위에 의존하면 안 됨)
  vendor/  ──✗──→ staging/   (외부가 내부에 의존하면 안 됨)
  pkg/A    ──✗──→ pkg/A      (자기 자신에 대한 순환 의존)
```

### 7.2 .import-restrictions 파일

각 패키지 디렉토리에 `.import-restrictions` 파일을 두어, 해당 패키지가 import할 수 있는 다른 패키지를 명시적으로 제한함.

**실제 예시 1**: `staging/src/k8s.io/apiserver/.import-restrictions`

```yaml
rules:
  # k8s.io/kubernetes를 import하는 것을 금지함
  # (apiserver가 루트 모듈에 의존하면 순환이 발생하기 때문임)
  - selectorRegexp: k8s[.]io/kubernetes
    forbiddenPrefixes:
      - ''        # 빈 문자열 = 모든 경로와 매칭됨
```

이 규칙은 `k8s.io/apiserver` staging 모듈이 `k8s.io/kubernetes`(루트 모듈)를 역으로 import하는 것을 금지함. 이렇게 해야 staging 모듈이 독립적으로 외부에 배포될 수 있음.

**실제 예시 2**: `staging/src/k8s.io/apiextensions-apiserver/pkg/apis/.import-restrictions`

```yaml
inverseRules:
  # 이 패키지를 import할 수 있는 곳을 제한함
  - selectorRegexp: k8s[.]io/apiextensions-apiserver
    allowedPrefixes:
      - ''                                    # 자기 자신은 허용
  - selectorRegexp: k8s[.]io/kubernetes/test
    allowedPrefixes:
      - k8s.io/kubernetes/test/e2e/apimachinery  # 테스트는 허용
  - selectorRegexp: k8s[.]io
    forbiddenPrefixes:
      - ''                                    # 그 외 k8s.io 패키지는 금지
```

`inverseRules`는 "이 패키지를 **누가** import할 수 있는가"를 제한하는 역방향 규칙임.

## 8. OWNERS 파일: 코드 소유권 관리

> 각 디렉토리에 OWNERS 파일을 두어 코드 리뷰/승인 책임을 분산함.

### 8.1 표준 형식 예시: `pkg/controller/OWNERS`

```yaml
approvers:          # PR 승인 권한을 가진 사람/팀
  - deads2k
  - derekwaynecarr
  - mikedanese
  - janetkuo
  - cheftako
  - andrewsykim
  - sig-apps-approvers    # 팀 단위 지정

reviewers:          # 리뷰 요청을 받는 사람/팀
  - andrewsykim
  - deads2k
  - cheftako
  - jefftree
  - sig-apps-reviewers

labels:             # PR에 자동 부착되는 라벨
  - sig/apps
```

### 8.2 필터 형식 예시: 루트 `OWNERS`

```yaml
filters:
  ".*":                           # 모든 파일에 적용
    reviewers:
      - dep-reviewers
      - sig-architecture-approvers
    approvers:
      - dep-approvers
    emeritus_approvers:           # 과거 승인자 (참고용)
      - bgrant0607
      - brendandburns

  "go\\.(mod|sum|work|work\\.sum)$":   # go.mod 등 의존성 파일
    approvers:
      - dep-approvers
    required_reviewers:           # 반드시 리뷰해야 하는 사람
      - kubernetes/dep-approvers
    labels:
      - area/dependency

  "metrics\\.go$":                # 메트릭 파일
    labels:
      - sig/instrumentation
```

**OWNERS 파일 필드 요약**:
- `approvers`: PR 승인(merge) 권한을 가진 사람 목록임
- `reviewers`: 리뷰 요청을 받는 사람 목록임
- `required_reviewers`: 반드시 리뷰해야 하는 사람임
- `emeritus_approvers`: 과거 승인자 목록 (참고용)임
- `labels`: PR에 자동 부착되는 라벨임
- `filters`: 정규식으로 파일별 다른 규칙을 적용함

## 9. 초보자를 위한 핵심 요약

### "의존성은 이렇게 흐름"

```
[가장 많이 의존되는 모듈 TOP 5]

1위: apimachinery  ← 거의 모든 모듈이 사용함 (API의 기본 타입)
2위: api           ← 리소스 타입을 정의함 (Pod, Service 등)
3위: client-go     ← API 서버와 통신하는 모든 컴포넌트가 사용함
4위: component-base← 공통 유틸리티 (FeatureGate, 로깅, 플래그 등)
5위: apiserver     ← API 서버 프레임워크 (Storage 인터페이스 등)
```

### "의존성 규칙 3가지"

1. **위에서 아래로만**: `cmd → pkg → staging → vendor` 방향으로만 의존할 수 있음
2. **순환 금지**: A→B→A 같은 순환 의존성은 절대 허용되지 않음 (`.import-restrictions`와 `verify-no-vendor-cycles.sh`로 강제함)
3. **etcd는 API 서버만**: etcd 클라이언트에 직접 의존하는 것은 apiserver 관련 코드만 허용됨
